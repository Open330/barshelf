# menubucket — 제품/UX 기획서 (R01)

> 관점: 제품 전략 + UX 디자인 | 작성일: 2026-07-07
> 한 줄 정의: **"메뉴바 하나에 담는 나만의 위젯 버킷"** — 스크립트/CLI/선언적 워크플로우로 만드는 네이티브 감성의 macOS 메뉴바 위젯 플랫폼.

---

## 0. 제품 비전 & 핵심 원칙

**비전**: 개발자·파워유저가 매일 확인하는 "작지만 소중한 정보"들이 브라우저 탭, 터미널 창, 수십 개의 메뉴바 아이콘에 흩어져 있다. menubucket은 이것을 **하나의 메뉴바 아이콘, 하나의 팝업** 안에 네이티브 품질로 모은다.

**3대 원칙**:
1. **One Icon, Many Widgets** — 메뉴바 오염 금지. 아이콘은 하나, 위젯은 무한.
2. **CLI is the API** — 이미 있는 도구(otpeek, aas, gh, kubectl…)가 그대로 데이터소스. 새 SDK 학습 강요 없음.
3. **Native or Nothing** — Electron 웹뷰 감성 배제. NSPopover/SwiftUI 수준의 반응성, 다크모드, 비브런시(vibrancy), SF Symbols.

## 2. 정보 구조 (IA) & 인터랙션 스펙

### 2.1 구조 개요

```
메뉴바 아이콘 (1개, 상태 배지 가능)
└─ 팝업 (NSPopover, 기본 360×480pt, 위젯에 따라 신축)
   ├─ [고정 영역] Pinned Row — 항상 최상단, 페이지 전환에도 유지
   ├─ [페이지 영역] Bucket(그룹) = 페이지 단위
   │    ├─ Bucket 1: "Work"   (CI, 캘린더, 서버)
   │    ├─ Bucket 2: "AI"     (aas usage, OTP)
   │    └─ Bucket 3: "Life"   (환율, IoT, Pomodoro)
   └─ [하단 바] 페이지 도트 · 검색(⌘F) · 설정(⚙) · 편집 모드
```

**설계 판단**:
- **Bucket = 페이지**. 작은 팝업에서 스크롤과 페이지를 동시에 쓰면 길을 잃는다. 세로 스크롤은 *한 Bucket 안에서만*, 가로 스와이프는 *Bucket 간 이동*으로 축을 분리한다.
- **Pin은 Bucket 위에 뜬다**. OTP처럼 어디서든 필요한 위젯은 핀 고정 → 모든 페이지 상단에 상주. 최대 2행 제한(팝업 잠식 방지).
- **검색은 위젯 검색 + 위젯 내 콘텐츠 검색을 통합**. `⌘F` → "aws" 입력 시 ① 위젯 이름 매칭 ② 각 위젯이 노출한 검색 가능 항목(OTP 계정명, 파일명 등) 매칭을 한 리스트로.

### 2.4 상태 & 엣지 케이스
- **로딩**: 위젯별 스켈레톤(shimmer). 팝업 전체를 블로킹하지 않음 — 마지막 캐시를 먼저 그리고 백그라운드 갱신 (stale-while-revalidate).
- **에러**: 위젯 카드가 접히며 한 줄 에러 + "로그 보기 / 재시도". 스크립트 stderr는 위젯 내 접이식 콘솔로.
- **빈 Bucket**: "+ 위젯 추가" 대형 타겟 + 갤러리 추천 3개.

## 3. 위젯 사이즈 클래스 & 레이아웃 시스템

iOS 위젯의 사이즈 클래스 멘탈 모델을 차용하되, 메뉴바 팝업에 맞게 4단계.

| 클래스 | 위치/크기 | 그리드 | 용도 | 예시 |
|---|---|---|---|---|
| **XS · Inline** | 메뉴바 자체, 텍스트/아이콘 | — | 상시 노출 1줄 상태 | `🍅 17:32`, `BTC 62.3K ▲`, CI 빨간 점 |
| **S · Tile** | 팝업 내 1×1 (2열 그리드의 반폭, 高 72pt) | 2열 | 단일 수치 + 라벨 + 1액션 | OTP 1계정, 다음 미팅, IoT 스위치 |
| **M · Card** | 팝업 전폭 ×1 (高 96–240pt, 콘텐츠 신축) | 1열 | 리스트/테이블/차트 | aas 테이블, 최근 파일, 클립보드 |
| **L · Panel** | 팝업이 확장된 전용 화면 (360→520pt 폭) | 단독 | 위젯 하나에 몰입 | 파일 브라우저 탐색, 로그 뷰어 |

**레이아웃 규칙**:
- 팝업 내부는 **2열 × 8pt 베이스 그리드**. S는 1칸, M은 2칸을 차지 — 계산 없는 자동 리플로우.
- 위젯 제작자는 콘텐츠(선언적 컴포넌트: `Label`, `Gauge`, `Table`, `List`, `Grid`, `Sparkline`, `Button`)만 선언하고, **각 사이즈 클래스별 표현을 옵션으로 제공** — iOS 위젯의 `supportedFamilies`처럼. 미지원 클래스는 시스템이 요약 폴백(제목+대표 수치) 자동 생성.
- **동일 위젯의 승격/강등**: XS(메뉴바) ↔ S ↔ M ↔ L은 같은 데이터의 다른 뷰. 핀치/더블클릭으로 사용자가 결정하며, 위젯 설정에 저장.

```
XS (메뉴바):   [ 🍎 …  🤖 71%  🍅 17:32  🔋 ]      ← inline 위젯 최대 3개 권장
S (타일):      ┌────────────┐                       M (카드):  ┌──────────────────────────┐
               │ 🤖 work-01 │                                  │ 🤖 aas usage             │
               │ 71% ▓▓▓░  │                                  │ work-01  1.2M  ▓▓▓░ 71% │
               │ 1.2M tok   │                                  │ personal 340K  ▓░░░ 22% │
               └────────────┘                                  │ team-x   2.9M  ▓▓▓▓ 94%⚠│
                                                               └──────────────────────────┘
L (패널):      팝업 확장 — 사이드 필터 + 상세 테이블 + 기간 스파크라인
```

## 4. 온보딩 & 위젯 설치 플로우

### 4.1 첫 실행 (60초 안에 "내 위젯"까지)

```

│ 시간입니다        │     │ ☐ 💱 환율 티커    │      │                  │

- 스타터 위젯은 **CLI 의존성 없는 것**으로 구성 → 빈 팝업을 절대 보여주지 않는다 (aha moment 선불 지급).
- CLI 필요한 위젯은 설치 시 의존성 자동 감지: `otpeek 미설치 → [brew install otpeek 실행] 버튼` 원클릭 해결.

```
- **설치 전 미리보기**: 샘플 데이터로 실제 렌더링을 보여줌 (스크린샷이 아니라 라이브 렌더). "메뉴바에서 어떻게 보일지"에 대한 불안 제거.
- **신뢰 UX**: 위젯이 요구하는 것(실행할 CLI, 네트워크 도메인, 갱신 주기)을 설치 시트에 명시 — App Store 프라이버시 라벨의 위젯판.

### 4.3 제작/편집 경험 — 이중 트랙

**트랙 A: 스크립트 (P1용)** — "CLI 출력 → 위젯" 3분 완성
```
┌─────────────────────────────────────────────────┐
│ 새 위젯: 스크립트                       [실행 ▶] │
├───────────────────────────┬─────────────────────┤
│ # widget.py               │  LIVE PREVIEW       │
│ out = run("aas usage --json")│ ┌────────────────┐│
│ rows = json.loads(out)    │  │ 🤖 aas usage    ││
│ return Table(             │  │ work-01 ▓▓▓ 71%││
│   rows, refresh="30s",    │  │ personal ▓ 22% ││
│   on_click=copy)          │  └────────────────┘│
│                           │  사이즈: [S][M][L]  │
├───────────────────────────┴─────────────────────┤
│ 💡 감지됨: JSON 배열 출력 → Table 자동 제안       │
└─────────────────────────────────────────────────┘
```
- 핵심은 **라이브 프리뷰 분할 뷰** + **출력 자동 감지**: 스크립트가 JSON/TSV를 뱉으면 Table/List 스캐폴드를 제안. "파싱 코드"가 아니라 "표현 선언"만 쓰게 한다.
- 외부 에디터 존중: 위젯은 폴더(`~/…/widgets/my-widget/`)이며 파일 변경 시 핫 리로드. VS Code로 편집하는 사람을 앱에 가두지 않는다.

**트랙 B: 선언적 워크플로우 (P3용)** — Shortcuts 스타일 블록
```
┌────────────────────────────────────┐
│ 새 위젯: 워크플로우                  │
│ ① [매 5분마다 ▾]                    │
│ ② [셸 실행: curl api.exchange...]   │
│ ③ [JSON에서 값 꺼내기: .rates.KRW]  │
│ ④ [표시: 인라인 텍스트 "₩{value}"]  │
│    └ 조건: value > 1400 → 🔴 강조   │
└────────────────────────────────────┘
```
- 트랙 B로 만든 위젯은 **"코드로 보기"로 트랙 A 스크립트로 export 가능** → 성장 경로 제공 (Shortcuts가 못 하는 것).

## 5. 경쟁 비교 & 포지셔닝

| 제품 | 취할 것 ✅ | 피할 것 ❌ |
|---|---|---|
| **xbar** | 플러그인 = 실행파일 하나라는 극단적 단순함, 출력 규약(stdout DSL)의 낮은 진입장벽 | 위젯 1개 = 메뉴바 아이콘 1개(메뉴바 오염), NSMenu 텍스트 나열의 빈약한 UI, 유지보수 침체 |
| **SwiftBar** | 네이티브 성능, SF Symbols/컬러 지원, 스트리밍 갱신 | 여전히 메뉴 항목 나열 패러다임 — 테이블/게이지/그리드 불가, 플러그인 발견성 부족 |
| **Raycast** | 스토어 큐레이션 품질, 원클릭 설치, 키보드 퍼스트, 브랜딩/DX 문서 수준 | 런처 중심(호출해야 보임 — 상시 glanceability 없음), React 전용 확장의 높은 진입장벽, 구독 피로감 |
| **Übersicht** | 데스크톱 상시 표시라는 glanceability, 자유로운 시각화 | 웹뷰 기반의 이질감·전력 소모, 데스크톱에 깔려 클릭 인터랙션이 사실상 없음 |
| **iOS Shortcuts** | 블록형 노코드 빌더, 갤러리와 공유 링크 배포 모델 | 디버깅 불가능한 블랙박스, 깊어지면 스파게티가 되는 블록 UX — export-to-code로 탈출구 제공 |

**포지셔닝 문장**:
> *"xbar의 개방성 × Raycast의 완성도 × iOS 위젯의 레이아웃 시스템 — 단, 메뉴바 아이콘은 단 하나."*

**차별화 축 3가지**:
1. **집약 (Aggregation)**: 경쟁자들은 위젯당 아이콘 하나이거나(xbar/SwiftBar) 호출형(Raycast). menubucket만이 "하나의 팝업, 여러 위젯, 그룹/페이지/핀" IA를 가짐.
2. **표현력 (Layout System)**: stdout 텍스트 나열이 아닌 선언적 컴포넌트 + 사이즈 클래스 — 메뉴바 생태계에 iOS 위젯급 레이아웃을 최초 도입.
3. **이중 트랙 제작**: 스크립트(개발자)와 블록(비개발자)이 같은 위젯 포맷으로 수렴 — 시장 양쪽을 하나의 생태계로.

## 7. 네이밍 & 브랜딩

### 후보 평가

| 이름 | 콘셉트 | 평가 |
|---|---|---|
| **menubucket** (현) | 메뉴바 + 버킷(담는다) | 직관적, 도메인/검색 유리. 발음이 다소 김 → 애칭 "**mbk**"/"버킷" 병용 |
| Bucketbar | 버킷+바 | 명료하나 xbar/SwiftBar 아류 인상 |
| Tray | 쟁반(담는 판) | 짧고 우아하나 일반명사라 검색·상표 불리 |
| Alcove | 벽감(작은 공간) | 감성적, 그러나 기능 전달력 약함 |

**결론: menubucket 유지.** 브랜드 시스템:
- **로고/아이콘**: 메뉴바 속 미니멀 버킷(🪣) 실루엣 — 위젯 상태에 따라 버킷에 "차오르는" 배지 애니메이션 (예: 알림 시 물방울).
- **CLI**: `mbk` — `mbk install`, `mbk new`, `mbk dev` (라이브 프리뷰 모드).
- **어휘 체계**: 위젯 그룹 = **Bucket** (제품명과 개념이 일치 — "Work 버킷에 담아두세요"), 위젯 단위 = **Drop**(선택적 애칭: "버킷에 드롭을 담는다").
- **톤**: 개발자에게는 정확하게(스펙 문서 품질), 마케팅은 가볍게 — 태그라인 후보:
  - *"One icon. All your widgets."*
  - *"Your menu bar, finally organized."*
  - 한국어: *"메뉴바, 이제 하나면 됩니다."*
